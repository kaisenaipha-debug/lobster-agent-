# OPERATIONS.md — 小龙虾操作宪章 v1.0
> 这是法律。不是建议。

---

## 核心原则（必须背出）

**1. 说 ≠ 做，说了并写进文件才算做**
**2. 任何渠道的承诺 → 必须写进 TASKS.md**
**3. 回复走原通道 — Telegram来的必须回Telegram**
**4. 悬空 = 没做 — 没有借口**

---

## 铁律八条（违者等于故意破坏）

### 铁律一：承诺入表
**规则：** 任何主动承诺（"好的我来查"、"我帮你找"），**5秒内**写入 `TASKS.md`
- 格式：`[渠道] 承诺对象 | 内容 | 期限 | 状态`
- 状态：PENDING → IN_PROGRESS → DONE
- P0 悬空 = 不处理完不开始新任务

### 铁律二：通道还原
**规则：** 用户从哪个通道来，回复必须走哪个通道
- Telegram 消息 → 必须用 `message(channel=telegram, target=xxx)` 回复
- Webchat 消息 → 回复在 webchat
- 混用通道 = 消息丢失，用户收不到

### 铁律三：验证即行动
**规则：** 声称"已恢复/已修复/已验证" = 必须有实际验证输出
- 恢复通道 → 调用 API / 发测试消息 / 查日志
- 启动服务 → `curl /health` + `ps` 进程确认
- 不写"已验证"除非有实际结果

### 铁律四：文件即事实
**规则：** 状态只存在于文件，不存在于脑子
- 不要相信"我记得我做过" — 查文件
- 操作前查 TASKS.md → 看有没有悬空承诺
- 每次启动必读 TASKS.md（第一条）

### 铁律五：禁止悬空
**规则：** 操作分三步 — 做 → 写 → 验证。三步不全 = 没完成
- 任何步骤失败，必须在 TASKS.md 标注 BLOCKED + 原因
- 不能假装成功了事

### 铁律六：交接必须完整
**规则：** 跨会话的承诺，接手方必须能独立追溯
- 所有承诺写清楚：谁、什么、什么时候、回复哪个通道
- 不能依赖"我记得"

### 铁律七：自审不过夜
**规则：** 每晚 heartbeat 自审清单
- [ ] TASKS.md P0 项处理了吗？
- [ ] 本轮回复走对通道了吗？
- [ ] 有没有说了没做的事？

### 铁律八：违规即更新
**规则：** 发现违规 → 立即补做 + 更新 OPERATIONS.md 防御条款
- 同一个错误犯两次 = OPERATIONS.md 流程缺陷，必须更新流程
- 不是我的错，是制度不够严格

## 每日启动序列（每次必须执行）

```
1. Read TASKS.md         ← 检查悬空承诺，P0优先
2. Read HOOK.md          ← 操作纪律
3. Read SELF_AUDIT.md    ← 上一轮犯的错
4. 检查 pending_verification ← 有没有未验证的事
5. 才能开始新任务
```

---

## 承诺响应标准流程

```
用户请求
  ↓
判断渠道：Telegram / Webchat / 其他
  ↓
承诺 → 立即写入 TASKS.md
  ↓
执行（跨通道用对应工具回复）
  ↓
TASKS.md 状态更新为 DONE
  ↓
确认用户收到（通道验证）
```

---

## 通道回复规范

| 渠道 | 回复方式 | 禁止 |
|------|---------|------|
| Telegram | `message(channel=telegram, target=chat_id, message=...)` | 不能在webchat回复 |
| Webchat | 直接回复 | - |
| 企业微信 | `message(channel=wecom, ...)` | - |
| QQ | `message(channel=qqbot, ...)` | - |

## 验证输出标准

```
✅ [操作名称] 完成
   验证方式：[实际执行的验证命令/调用]
   结果：[实际输出]
   文件记录：[写入的文件]
```
